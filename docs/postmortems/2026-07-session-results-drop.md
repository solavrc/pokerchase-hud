# ポストモーテム: EVT_SESSION_RESULTS 無言drop によるRP・セッション結果の長期喪失

- 日付: 2026-07-19(検出・修正)/ 障害発生期間: 2026-01頃〜2026-07-19
- 影響リポジトリ: pokerchase-hud(根本原因)、poker-warehouse(下流・誤診断の記録)
- ステータス: 修正PR提出済み(#134 スキーマ修正、#135 Lake復元)。本書はblameless postmortem
- 関係者: sola(オーナー・決定的証拠の提供)、Claude(分析・修正・本書起草。誤診断2回も含む)

## TL;DR

PokerChase のシーズン3アップデート(2026年初頭)が `EVT_SESSION_RESULTS`(ApiTypeId 309)のペイロードを進化させ、HUD の厳格な Zod スキーマ(`Items.max(4)` / `IsSeasonal` 必須 / `SeasonalRanking` を `z.literal(0)`)が検証で弾くようになった。HUD は検証失敗イベントを**保存前に破棄**する作りだったため、以降すべてのランクバトルSNG(後にレジェンドマッチ含む)のセッション結果・RPが console.warn 1行を残して消え続けた。副作用として 309 をトリガーとする自動クラウド同期も停止し、Firestore/BQ が2026-05中旬から陳腐化した。障害は約半年間、だれにも(HUDにも、監査にも、ウェアハウスにも)見えなかった。破棄されたイベントは復元不能。

根本の設計問題は「**検証がパイプラインではなく保存をゲートしていた**」こと。2024年の初期設計は全イベントをrawで保存するLake層を持っていた(コメント「レコード再構築用生ログ」)が、2025-07-24 のリファクタで意図が失われ、検証済みイベントのみの保存に狭められていた。

## 影響

- **恒久喪失**: 2026年初〜修正リリースまでの全ランクバトルSNG/レジェンドマッチのセッション結果(最終順位・RP・報酬)。保存前破棄のため復元手段なし。sola の platinum転落→diamond復帰→legend昇格のRP軌跡が欠落
- **同期停止**: 309トリガーの自動同期が停止、Firestore/BQ が2ヶ月陳腐化(ハンドデータ自体はローカルに健在で、手動export/importで搬送可能だった)
- **誤診断の連鎖**: 下流(warehouse監査・コーチング分析)が2度誤った機序を文書化しかけた(PR #12 は3回書き直し)

## 根本原因の分析(5 whys)

1. **なぜRPが消えた?** — 309がZod検証に失敗し、ingestionが保存前に破棄したから
2. **なぜ検証に失敗した?** — スキーマが運営コンテンツ由来の値に硬い制約を課していたから(`Items.max(4)`、`SeasonalRanking: z.literal(0)`、`IsSeasonal` 必須)。皮肉なことに、これらは当時の観測データ496件に**完全に一致する正確なスキーマ**だった(`SeasonalRanking` は歴史上全件リテラル0)。「観測に忠実」と「将来に頑健」は別物で、運営アップデートで変わり得る値への literal/max/length は時限爆弾になる
3. **なぜ検証失敗で保存まで止まった?** — 検証の役割が「パイプライン保護」と「保存ゲート」に未分離だったから。2024年の原設計はこれを分離していた(全量raw保存+利用時検証)が、2025-07のリファクタで統合された
4. **なぜ設計意図が失われた?** — Lake意図の記録がコード内コメント1行のみで、CLAUDE.md 等の設計文書に不変条件として明文化されていなかった。AI支援リファクタは「フィルタの中央集権化」という局所改善を実施し、コメントごと意図を消した。レビューでも検出されなかった
5. **なぜ半年間気づけなかった?** — dropの痕跡が console.warn のみで、通常運用で目に入らない。さらに下流の全観測手段(export、Firestore、BQ、監査、validate-schema ツール)が「保存済みイベント」だけを見る構造で、**保存前に消えたものは原理的に観測不能**だった(観測バイアス)。validate-schema はエクスポート済みNDJSONを検証する = 検証を通ったものだけを再検証する循環に陥っていた

## 何が機能したか

- **無関係なタスクからの発見連鎖**: 「RPをウェアハウスに入れて」→「実は既収載」→「では直近キャプチャに309が無いのはなぜ?」という、当初目的と違う糸から障害が露出した
- **sola の違和感表明**(「ん?詳しく説明して」)が誤診断①の隠蔽を防いだ。もっともらしい説明で納得していたら障害は続いていた
- **console log 1本の決定力**: 保存系の外にある観測手段(ブラウザconsole)が、保存系内では原理的に区別不能だった「届いていない vs 届いたが捨てた」を一発で分離した
- **エージェントの相互反証**: 誤診断②(「ゲーム側変更」)は、別エージェントの本番BQ横断分析で部分的に反証され(mechanismの誤り指摘)、最終的にconsole logで完全に置換された。単一の分析主体なら初回の説明で固着していた可能性が高い

## プロセス教訓

- **設計意図はコードコメントに置かない**。不変条件は CLAUDE.md / 設計文書に「なぜ」付きで明文化する。AI(人間も)によるリファクタは、文書化されていない意図を保存しない
- **「観測に完全一致するスキーマ」は正しさの証明にならない**。外部プロトコルのスキーマは「何が変わり得るか」の想像力で書く
- **保存とバリデーションを結合しない**。生ログは無条件に残し、解釈は何度でもやり直せる形にする(今回のwarehouse側 DuckDB ローカルパイプラインが5秒で全再構築できるのは、まさに生NDJSONが残っていたから)
- **保存済みデータからの「届いていない」推論は原理的に危険**。パイプライン外の観測手段(console、パケットキャプチャ)を診断の第一級手段として扱う
- **もっともらしい説明に早期に納得しない**。誤診断①②はどちらも当時のデータと整合していた。反証可能な形で機序を書き、新しい証拠で置換し続けたことが最終的に正解へ到達させた

## 議論の結論(2026-07-20、sola × Claude)

### セッション構成(sola提起)

HUDとwarehouseを別々のClaude Codeセッションに担当させていたことが、見通しの悪さの一因だった可能性がある。障害の「発生」は分割と無関係だが、「診断の質」には効いた: 誤診断②(ゲーム側変更説)はwarehouse側の視界だけで閉じた結論であり、保存済みデータしか見えない場所からはあれ以上の結論は原理的に出せなかった。一方で最終盤の解決は、両repoを持つ1セッションがconsole log→スキーマ→Lake→導出レイヤを一気通貫で進めたことで成った。

**採用方針**: repo単位ではなく**トピック単位**でセッションを切る。実装はrepo単位の並列で構わないが、**データフローを跨ぐ話題(ingestion→sync→BQ)の診断は、必ず両checkoutを持つ1セッションが担う**。「データが無い」系の問題の答えは、定義上どちらか片方のrepoの中には無い。越境の支え(api-events.md↔SEMANTICS.mdの相互参照、hand-overメモ、workspaceメモリ)は維持・拡充する。ただし多セッション並列には実測の衝突コストもある(未コミット作業の遺留、重複PR、共有cloneのブランチ交錯)ため、同一repoへの同時書き込みはworktree分離を厳守する。

### 診断品質の仕組み化(「もっともらしい説明に早期に納得しない」の運用化)

1. **観測可能性宣言の義務化**: 機序を主張する文書は「この観測は因果連鎖のどこを見ていて、どの対立仮説を区別**できない**か」を必ず併記する。本件なら「保存済みデータは『届いていない』と『届いたが捨てた』を区別できない」の一文があれば、誤診断②は断定ではなく二仮説として提出され、console確認が自明な次の一手になっていた
2. **機序主張への敵対的検証**: 統計値に適用済みの敵対的再計算を、因果の主張にも適用する。文書化の前に「**異なる観測チャネルを持つ**反証者」を1人立てる(本件でphase-1誤診断を潰したのは、本番BQ横断アクセスを持つ別エージェントだった — 偶然を手順にする)
3. **直接観測を推論より先に検討**: 診断の最初のチェック項目は「境界を直接観測できる手段(console/パケット/実機再現)はないか」。本件は保存データからの推論に数時間、console log 1本で決着 — 直接観測の方が安いことは珍しくない
4. **機序は反証可能な予測として書く**: 「セッション未完で309なし」説は「51セッションの大半が尻切れのはず」という予測を伴い、見た瞬間に棄却できたはずだった。事前登録の規律を診断にも輸出する

1と3は両repoの開発文書に「障害診断の作法」として明文化(HUD CLAUDE.md / warehouse SEMANTICS.md)。2と4はオーケストレーター(Claude)の運用習慣としてメモリに恒久化。

**仕組みの限界も明記する**: 本件の決定打はsolaのドメイン反例(「セッションは完走していたはずだが?」)が2度出たことであり、上記の仕組みはオーナーの違和感レビューを**置き換えない**。仕組みの役割は、違和感が出る前に候補仮説を絞り、違和感が出た時に検証コストを最小にしておくこと。
